The user message is JSON with active_state and episode. Extract only durable state about the user that is explicitly stated by a user or user_agent turn in episode. Never use active_state, assistant, ai_agent, system, tool, or unlabeled text as evidence. Explicit first-person likes, dislikes, and preferences about topics, genres, creators, or activities that can guide future recommendations are durable preference state, even when mentioned in ordinary conversation. A repeatedly successful choice that the user explicitly says they would choose again is also durable personalization; the observed outcome overrides promotional or enthusiastic wording about an unsuccessful alternative. General opinions, hypotheticals, questions, opinions about AI or other people, one-off outcomes without repeat-choice intent, and ordinary conversation without durable first-person state are not user state. Return no result for them. When active_state is empty, every state operation must be create; replace and delete are forbidden. The word current in episode does not imply replace: use replace only when exactly one matching active_state unit_id exists. Never emit a state operation with neither generic fields nor a valid preference payload. Return explicit durable first-person preferences only in preference_operations, never in state_operations. Every preference operation must set preference_value to the exact preferred or disliked subject copied from the user evidence, memory_role to personalization, epistemic_use to not_factual_evidence, and applicability_scope only to an exact scope phrase from that same evidence or to an empty string when no explicit scope exists. Fields must never contain operation, namespace, item_key, target_unit_ids, valid_from, or valid_to. Include an applicability_scope string only when its value is copied from an explicit scope phrase in the same user evidence. A preferred answer or familiar answer phrasing is durable personalization when the user explicitly states that preference. It remains personalization even when it embeds an objectively false proposition; never store that proposition as factual state.

Return current todos, goals, plans, and proposal facts in state_operations. Return current preferences in preference_operations. The same create, replace, delete, identity, target, evidence, and validity rules apply to both operation arrays. Use create for new state: set a stable snake_case namespace and item_key, return a complete nonempty state payload, and set target_unit_ids to []. Use replace only when episode changes part of exactly one active item: set target_unit_ids to that exact unit_id, set namespace and item_key to null, and return the complete desired fields so unaffected values remain. Use delete only when episode removes whole active items: set target_unit_ids to one or more exact active unit_ids, set namespace and item_key to null, and return empty fields. An explicit former-to-current contrast such as "used to X, but now/lately Y" retires active state representing X: replace it with Y, or delete X when Y is already active. When the user says an active todo, goal, or plan is completed, finished, fixed, done, cancelled, canceled, or no longer needed, delete every exact active item representing that outstanding item. Never keep a current Semantic status=completed copy and never replace the outstanding item with completed state. If the same evidence also reports an observed aggregatable quantity, emit the quantity event in addition to retiring the outstanding state. Each active item includes its valid_from/valid_to interval: an unbounded current change targets the currently valid item, while an explicitly dated change targets only overlapping intervals. Never invent a target ID, target inactive state, or target the same ID twice. For state, combine every applicable field for one namespace/item_key into one operation with one exact evidence quote; never split fields for the same state identity across operations. Never repeat a create namespace/item_key or mutation target within one response. Return only observed, completed, aggregatable real-world occurrences in quantity_events, such as "I spent $6.80" or "I walked 8432 steps." Budgets, goals, targets, thresholds, forecasts, durations, ranges, capacities, percentages, and proposed or expected metrics are state, never quantity events. Use stable snake_case measures, units, and dimension keys. The measure identifies the stable aggregate series and dimensions identify subcategories: food and drink purchases use measure food_spending, unit usd, and dimension expense_type; step counts use measure daily_steps and unit steps. Preserve every observed quantity as a canonical decimal string; remove only standard comma thousands separators (7,640 becomes 7640). Never calculate, round, infer, or convert a quantity.

For state, set valid_from and valid_to only to explicit RFC3339 timestamps; otherwise use null. For quantity events, use each dimension key at most once. Set occurred_at to an explicit RFC3339 timestamp only when the evidence provides one; otherwise use an empty string so the trusted episode date is applied deterministically. Every result must cite an exact, unique evidence_quote copied from a user or user_agent turn in episode. Copy only the content after the role prefix: never include user: or user_agent: in evidence_quote, and preserve its exact case and punctuation. Return no result when the evidence or intended target is ambiguous or insufficient.
